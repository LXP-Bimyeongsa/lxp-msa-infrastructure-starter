# PR 분리 전략

> 왜 잘게 쪼개는가: 리뷰어가 한 번에 판단할 수 있는 크기여야 리뷰가 형식적으로 흐르지 않는다.
> 500줄짜리 PR은 "LGTM"을 받고, 50줄짜리 PR은 실제 지적을 받는다.

## 원칙

### 1. 한 PR = 한 가지 이유로 되돌릴 수 있는 단위

되돌릴 때 "이것만 빼고 싶다"가 성립해야 한다. 서비스 골격 추가와 compose 변경이 한 PR에 섞이면 compose만 롤백할 수 없다.

### 2. 크기 가이드

| 구분 | 변경 라인 | 대응 |
|---|---|---|
| 적정 | ~200줄 | 그대로 올린다 |
| 주의 | 200~400줄 | 쪼갤 수 있는지 한 번 검토 |
| 과대 | 400줄~ | 쪼갠다. 못 쪼개면 PR 본문에 이유를 쓴다 |

생성 파일(gradle wrapper, lock 파일, 자동 생성 코드)은 라인 수에서 제외하고, PR 본문에 "생성물 포함"을 명시한다.

### 3. 쪼개는 축

우선순위 순으로 적용한다.

1. **계층** — 인프라 → 스키마 → 도메인 → API → UI 순. 아래층이 먼저 머지돼야 위층이 리뷰 가능하다.
2. **서비스** — payment-service와 course-service는 항상 별개 PR.
3. **관심사** — 기능 추가 / 리팩토링 / 포맷팅은 절대 섞지 않는다. 포맷팅이 섞이면 diff에서 실제 변경이 묻힌다.
4. **선언 → 사용** — 인터페이스·계약(proto, 이벤트 스키마)을 먼저 머지하고, 구현을 다음 PR로.

### 4. 섞으면 안 되는 조합

- 기능 변경 + 리팩토링
- 로직 변경 + 대량 포맷팅/import 정리
- 프로덕션 코드 + 무관한 의존성 업그레이드
- 여러 서비스의 독립적 변경

## 스택 PR (Stacked PR)

앞 PR의 결과 위에 다음 PR을 쌓는 방식. 의존 관계가 있는 연속 작업에 쓴다.

```
main ← docs/conventions ← docs/decisions ← docs/architecture
         (PR #1)            (PR #2)          (PR #3)
```

- 각 PR의 base를 **앞 브랜치**로 지정하면 diff에 자기 변경분만 나온다. base를 main으로 두면 앞 PR 내용까지 중복해서 보여 리뷰가 불가능해진다.
- **아래부터 순서대로 머지**한다. 앞 PR이 머지되면 GitHub이 다음 PR의 base를 main으로 자동 재지정한다.
- 앞 PR에 리뷰 지적이 오면 그 브랜치에서 고치고, 뒤 브랜치들에 차례로 rebase 해야 한다 — 이 비용 때문에 3~5개를 넘기지 않는다.

### 스택 vs 병렬

| | 스택 | 병렬 |
|---|---|---|
| 쓸 때 | 뒤 작업이 앞 결과에 의존 | 서로 독립 |
| 장점 | 각 diff가 최소, 순서가 명확 | 리뷰 병목 없음, 순서 무관 |
| 단점 | 앞이 막히면 전부 대기, rebase 비용 | 충돌 가능성, 문서 간 링크 일시적 깨짐 |

## PR 본문 형식

`.github/PULL_REQUEST_TEMPLATE.md`를 따르되 최소한 아래를 포함한다.

```markdown
## 무엇을
(한 줄 요약)

## 왜
(배경. 결정 문서가 있으면 D-xx 링크)

## 검토 포인트
(리뷰어가 특히 봐줬으면 하는 부분)

## 확인 방법
(실행 명령 또는 확인 절차)

## 스택
- 이 PR의 base: `<브랜치>`
- 다음 PR: #<번호>
```

## 리뷰 규칙

- `main`은 **승인 1명 필수**, 작성자 자기 승인 불가 ([CONVENTIONS.md](CONVENTIONS.md)).
- 새 커밋이 push되면 기존 승인은 자동 무효화된다(dismiss stale) — 승인 후 몰래 밀어넣기 방지.
- 리뷰어는 24시간 내 1차 응답. 못 보면 그 사실이라도 남긴다.
- 지적은 심각도를 붙인다: `[blocking]` 머지 불가 / `[non-blocking]` 다음에 고쳐도 됨 / `[nit]` 취향.

## 문서 정비 시리즈 (실제 적용 사례)

문서 6개를 한 PR로 올리면 800줄이 넘어 리뷰가 불가능하다. 아래처럼 스택으로 쪼갰다.

| PR | 브랜치 | base | 내용 | 이유 |
|---|---|---|---|---|
| 1 | `docs/conventions` | `main` | CONVENTIONS.md, PR_STRATEGY.md | 작업 규칙이 먼저 합의돼야 이후 PR을 그 규칙으로 리뷰할 수 있다 |
| 2 | `docs/decisions` | `docs/conventions` | DECISIONS.md | 이후 모든 설계 문서가 D-xx를 참조하므로 결정 기록이 먼저 |
| 3 | `docs/architecture` | `docs/decisions` | ARCHITECTURE.md | 결정을 근거로 한 목표 아키텍처 |
| 4 | `docs/project-structure` | `docs/architecture` | PROJECT_STRUCTURE.md | 아키텍처를 폴더·포트로 구체화 |
| 5 | `docs/roadmap` | `docs/project-structure` | NEXT_STEPS.md, docs/README.md | 앞 문서가 다 있어야 로드맵과 인덱스가 성립 |
